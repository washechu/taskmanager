# Уведомления через Telegram-бот — дизайн-док

**Статус:** дизайн утверждён (§11 закрыт). Готов к реализации PR α.
**Автор:** Никита (продукт) + Claude (архитектура).
**Решение по push:** Web Push отвергнут как нерабочий на iOS PWA и оверкилл для двух пользователей. Telegram даёт инлайн-кнопки в самом уведомлении и стабильно работает везде.

---

## 1. Цели

1. Превратить трекер из пассивного («открой и посмотри») в активный («дёрнул в нужный момент»).
2. Дать механику **предложения** для семейных задач: «давай в субботу в кино» → ✅/🤔/❌.
3. Без отдельного сервера. Всё ложится в текущий стек: Supabase + Vercel.

## 2. Принципы

- **Не шумим.** Меньше = лучше. Если можно не слать — не шлём.
- **Каждое сообщение — actionable.** Либо в нём кнопка, либо открыть/отметить можно прямо из приложения за один тап.
- **Расписание ≠ события.** Расписание — два сообщения в день максимум. События — реакция на действия партнёра, штучные.
- **Privacy.** Без участия Telegram-API сам бот ничего не знает о задачах. Текст формируется на нашей стороне, шлётся одной командой.

## 3. Архитектура

```
┌──────────────────┐    realtime    ┌──────────────┐
│   PWA (Next.js)  │ ◀────────────▶ │   Supabase   │
└────────┬─────────┘                └──────┬───────┘
         │                                 │
         │                       ┌─────────┴──────────┐
         │                       │                    │
         │                ┌──────▼──────┐     ┌──────▼─────┐
         │                │  pg_cron    │     │ DB-trigger │
         │                │ (расписание)│     │  (события) │
         │                └──────┬──────┘     └──────┬─────┘
         │                       │                   │
         │                       └──────┬────────────┘
         │                              │ pg_net.http_post
         │                              ▼
         │                  ┌─────────────────────────┐
         │                  │ Edge Function           │
         │                  │ supabase/functions/     │
         │                  │   send-telegram         │
         │                  └────────┬────────────────┘
         │                           │ Bot API
         │                           ▼
         │                  ┌──────────────────┐
         └────api/telegram◀─│   Telegram Bot   │
              (webhook)     └──────────────────┘
                                ▲       │
                          /start, ✅    push
                                │       ▼
                            пользователь
```

**Компоненты:**

| Что | Где живёт | Зачем |
|---|---|---|
| `send-telegram` | Supabase Edge Function (Deno) | Единая точка отправки. Достаёт `chat_id` из `telegram_links`, шлёт Bot API. |
| `pg_cron` | Supabase | Утренний и вечерний джобы → вызывают SQL-функцию `send_*` → `pg_net` дёргает Edge Function. |
| DB-триггер | Supabase | На `tasks` / `comments` ловит изменения, формирует событие, шлёт через `pg_net`. |
| Webhook | Vercel API route (`app/api/telegram/webhook/route.ts`) | Принимает `/start` и `callback_query` (кнопки) от Telegram. |
| Бот | @BotFather | Сам пользователь создаёт через @BotFather, токен в env. |

**Почему именно так:**
- pg_cron + DB-триггеры + pg_net позволяют **расписание и реакцию на события вообще без всегда-включённого сервера**. Vercel API route нужен только как обработчик входящих webhook'ов от Telegram — это тонкий слой.
- Edge Function для отправки — потому что SQL-функциям неудобно собирать markdown-сообщения и inline-клавиатуру, а в Deno это пара строк.
- Один источник истины для текстов сообщений — в Edge Function. Триггер передаёт только тип события и id; Edge Function запрашивает детали из БД и форматирует.

## 4. Модель данных (миграция 009)

```sql
-- Привязка Telegram-чата к пользователю системы.
create table telegram_links (
  assignee   text primary key check (assignee in ('nick', 'galya')),
  chat_id    bigint not null,
  username   text,                       -- для удобства, не обязательно
  created_at timestamptz default now()
);

-- Механика предложений на семейных задачах.
alter table tasks
  add column if not exists invited_by text
    check (invited_by in ('nick', 'galya')),  -- кто предложил (= создатель семейной)
  add column if not exists invite_status text not null default 'none'
    check (invite_status in ('none','pending','accepted','tentative','declined'));

-- Лог отправок (для диагностики «бот молчит»). Без ретраев — см. §11.8.
create table telegram_log (
  id         uuid primary key default gen_random_uuid(),
  assignee   text,                  -- кому слали (может быть null если массово)
  kind       text not null,         -- 'morning' | 'evening' | 'invite' | 'invite_reply' | 'comment' | 'done' | 'test'
  ok         boolean not null,      -- успех доставки (Telegram вернул 200)
  error      text,                  -- тело ошибки, если ok=false
  created_at timestamptz default now()
);
```

**Семантика `invite_status`:**

| Значение | Когда |
|---|---|
| `none` | Задача не «предложение». Все личные задачи; семейные с одним участником; задачи, созданные до миграции. |
| `pending` | Создана как семейная с двумя участниками. `invited_by = создатель`. Второй должен ответить. |
| `accepted` | Второй принял. Задача движется как обычная. |
| `tentative` | Второй ответил «Думаю». Задача активна, но статус-сигнал — «не финально». |
| `declined` | Второй отклонил. Задача переходит к одному `invited_by` (`assignees = [invited_by]`), `invite_status` → `none` — становится обычной личной задачей предложившего (решение §11.1, вариант **a**). |

**Когда `pending` создаётся автоматически:**
- `category='family'` И `assignees.length === 2`.
- Личные задачи и family-с-одним не превращаются в предложение.

**Транзишен при ответе:**
- `pending` → `accepted` / `tentative` / `declined` (только второй участник может ответить, не предложивший).
- **Переключение между финальными разрешено** (§11.2): `accepted` ↔ `tentative` ↔ `declined` можно менять сколько угодно. Назад в `pending` — нельзя. Жизнь меняется — план тоже.
  - Особый случай: если ответ был `declined` (задача стала личной у `invited_by`) и второй передумал — он уже не видит её как предложение. Поэтому переключение работает, пока `invite_status` ещё не `declined`-финализирован в `none`. После полного отказа — только новое предложение. (Т.е. «передумать» можно из `accepted`/`tentative`; из «отклонил» — нет, т.к. задача уже ушла из общего поля.)

## 5. Расписание

Два сообщения в день. Время — локальное (МСК по умолчанию; в pg_cron job хранится как UTC + timezone-математика). Каждое сообщение — **персонализированное на пользователя** (две доставки в каждом джобе, если у обоих заведены `telegram_links`).

### 5.1. 🌅 Утренний дайджест — 08:00

**Когда:** ежедневно, 08:00.
**Кому:** каждому пользователю с `telegram_links`.
**Что:** короткая сводка дня с действующими кнопками.

**Содержимое (динамическое):**
- Просрочка: количество.
- Задачи на сегодня: список (priority → text).
- Привычки на сегодня: список (только незавершённые на момент рассылки).

**Skip-условие:** если у пользователя на сегодня вообще ничего (нет просрочки, нет задач, нет привычек) — **не шлём**. Cпокойный день не дёргаем.

**Пример сообщения:**
```
🌅 Доброе утро, Никита!

🔥 Просрочено: 2
   • Записаться к зубному (28 мая)
   • Купить молоко (вчера)

⚠️ На сегодня (3):
   • Высокий — Сделать отчёт по Q2
   • Средний — Записаться к мануальщику
   • Низкий — Подобрать кронштейн для полки

🔁 Привычки (0/4):
   • Английский
   • Зарядка
   • Пить воду
   • Прогулка

[Открыть Сегодня]
```

Inline-кнопка `Открыть Сегодня` — `web_app` button с URL прод-приложения `/today` (Telegram открывает PWA в мини-приложении или браузере).

### 5.2. 🔁 Вечернее напоминание о привычках — 21:00

**Когда:** ежедневно, 21:00.
**Кому:** каждому пользователю с `telegram_links`.
**Что:** список незавершённых на сегодня привычек.

**Skip-условие:** все запланированные привычки на сегодня отмечены — **не шлём**.

**Пример:**
```
🔁 Не забудь отметить:

   • Английский
   • Прогулка

День ещё не закончился — и привычки тоже.

[Открыть Привычки]
```

### 5.3. Технически

- **pg_cron job:** `0 5 * * *` (08:00 МСК = 05:00 UTC) для утра, `0 18 * * *` для вечера.
- Job вызывает SQL-функцию `send_morning_digest()` / `send_evening_habits()`.
- Функция итерирует по `telegram_links`, собирает данные SELECT'ами, через `pg_net.http_post` дёргает Edge Function `send-telegram` с уже отформатированным текстом + кнопкой.

> Если когда-нибудь захочется индивидуальное время — добавим колонку `notify_time` в `telegram_links` и job будет звать функцию с параметром «отправь тем, у кого `notify_time = current minute`». Сейчас 08:00/21:00 захардкожено.

## 6. События

Реактивно — DB-триггеры на изменения в таблицах. **Всё, что выбрал ты, оставлено.**

### 6.1. 👋 Тебе предложили семейную задачу

**Когда:** `tasks.invite_status` стал `'pending'` (insert или transition из `'none'`).
**Кому:** второму участнику (тому, кто не `invited_by`).

**Пример:**
```
👋 Никита предложил тебе задачу

🎬 Кино в субботу
   📅 Дедлайн: 7 июня

«Премьера нового Дюна»

[✅ Принять]  [🤔 Думаю]  [❌ Отклонить]
```

Кнопки — inline-keyboard с `callback_data` вида `invite:<task_id>:<action>` (`accept` / `tentative` / `decline`).

### 6.2. ✅/🤔/❌ Ответ на твоё предложение

**Когда:** `tasks.invite_status` перешёл из `'pending'` в финальное.
**Кому:** инициатору (`invited_by`).

**Пример (принято):**
```
✅ Галочка приняла твоё предложение

🎬 Кино в субботу
```

Без кнопок — просто info-сообщение.

> **Audit-коммент (§11.6):** при ответе на предложение в ленту задачи пишется audit-строка: «Галочка приняла предложение» / «Галочка ответила: думаю» / «Галочка отклонила предложение». История переговоров видна в карточке. Пишется тем же механизмом, что и прочий audit (`kind='audit'`), автор — ответивший.

### 6.3. 💬 Новый комментарий

**Когда:** insert в `comments` с `kind = 'user'` (audit-комментарии не шлём).
**Кому:** всем участникам задачи, **кроме автора комментария**.

**Пример:**
```
💬 Галочка прокомментировала задачу

🎬 Кино в субботу

«Давай в воскресенье вечером — в субботу не получится»

[Открыть]
```

Inline-кнопка `Открыть` ведёт в `/tasks?open=<id>`.

### 6.4. ✔️ Партнёр выполнил общую задачу

**Когда:** `tasks.status` перешёл в `'done'`. И:
- задача семейная (`category='family'`)
- assignees.length ≥ 2
- меняющий — не я (т.е. в audit-комментарии author ≠ me)

**Кому:** всем участникам, кроме того, кто закрыл.

**Пример:**
```
✔️ Галочка закрыла общую задачу

🏠 Записать чарли в груминг
```

Без кнопок.

### 6.5. Технически

- Триггеры на `tasks` (AFTER UPDATE, AFTER INSERT) и на `comments` (AFTER INSERT).
- Триггеры — НЕ блокируют транзакцию: вызов `pg_net.http_post` асинхронный (он только кладёт в очередь, фактическая HTTP-отправка позже). Это правильно: пользователь не должен ждать Telegram при сохранении.
- Дедупликация: в триггере проверяем `OLD.status != NEW.status` и т.п., чтобы не слать на каждый чих.
- Если у получателя нет `telegram_links` — функция `send-telegram` молча ничего не делает.

## 7. Telegram-формат и кнопки

- **Markdown:** Telegram MarkdownV2 (требует экранирования `.`, `(`, `)`, `-`, и т.п.). Чтобы не возиться — шлём в HTML mode (`parse_mode: 'HTML'`), он лояльнее. Жирный `<b>`, курсив `<i>`.
- **Inline keyboards:** через `reply_markup.inline_keyboard`.
- **Web App buttons:** позволяют открыть страницу нашего PWA внутри Telegram. Используем для `[Открыть Сегодня]` и `[Открыть]`.
- **Длина:** сообщение Telegram до 4096 символов. Утренний дайджест может разрастись — обрезаем списки до N=10, добавляем `…ещё M`.

**Шаблон callback_data:**
```
invite:<task_id>:accept
invite:<task_id>:tentative
invite:<task_id>:decline
```
≤ 64 байт — telegram-ограничение. UUID занимает 36 — влезаем.

## 8. Webhook-контракт

`POST /api/telegram/webhook` (Vercel route, `app/api/telegram/webhook/route.ts`).

Принимаемые типы апдейтов:
1. **`message` с `text = '/start'`:** регистрирует `chat_id` → `telegram_links` для одного из двух пользователей.
   - Чтобы понять, кто Никита, а кто Галочка: бот отвечает «Привет! Кто ты?» с двумя inline-кнопками. Пользователь нажимает — `callback_query` сохраняет привязку.
   - Альтернатива: захардкодить два `username` в env и сравнивать с `message.from.username`. Чище, потому что без выбора. Идём этим путём; кнопочный fallback — если username не задан.
2. **`callback_query` с `data` начинающимся на `invite:`:** обновляет `tasks.invite_status`, шлёт ответ-сообщение «✅ Принято» (или редактирует исходное), генерирует событие §6.2.
3. **`callback_query` с `data = 'link:<assignee>'`:** альтернатива п.1 — пользователь сам выбирает, кто он.
4. **Игнорируем всё остальное** (вернуть 200 OK).

**Безопасность:**
- Telegram `setWebhook` поддерживает `secret_token` — Telegram кладёт его в header `X-Telegram-Bot-Api-Secret-Token`. Webhook валидирует. Если не совпало — 403.
- Никаких других данных у бота нет; даже если кто-то узнает URL — без token'а ничего не сделает.

## 9. Setup (один раз, руками)

Перед запуском фичи нужно:

1. **Создать бота:** в Telegram написать `@BotFather` → `/newbot` → имя → получить `TELEGRAM_BOT_TOKEN`.
2. **Сгенерировать webhook secret:** случайная строка, напр. `openssl rand -hex 32`.
3. **Положить в Vercel env:**
   - `TELEGRAM_BOT_TOKEN`
   - `TELEGRAM_WEBHOOK_SECRET`
4. **Положить в Supabase secrets:**
   - `TELEGRAM_BOT_TOKEN` (для Edge Function)
5. **Зарегистрировать webhook:**
   ```bash
   curl -X POST "https://api.telegram.org/bot<TOKEN>/setWebhook" \
     -d "url=https://taskmanager-sooty-phi-79.vercel.app/api/telegram/webhook" \
     -d "secret_token=<SECRET>"
   ```
6. **Применить миграцию 009** в Supabase SQL Editor.
7. **Создать pg_cron jobs** (после фазы β).
8. **Каждый пользователь** заходит в бота → `/start` → привязка.

## 11. Решения (зафиксировано)

1. **Поведение при `declined`** — ✅ вариант **(a)**: задача переходит к одному `invited_by` (`assignees = [invited_by]`), `invite_status` → `none`, становится обычной личной задачей предложившего.
2. **Изменение ответа** — ✅ **разрешено** переключаться `accepted` ↔ `tentative`, назад в `pending` нельзя. `declined` — односторонний выход (задача уходит из общего поля, передумать = новое предложение). См. §4.
3. **Контр-предложение** (отклонить с датой/причиной) — ⏸️ отложено на будущую фазу, не входит в γ.
4. **Время дайджестов** — ✅ захардкожено 08:00 / 21:00 МСК. Per-user `notify_time` — потом, если понадобится.
5. **Привычки в утреннем дайджесте** — ✅ общий список, без разделения daily/weekdays/monthdays.
6. **Audit-коммент при ответе** — ✅ нужен. Пишется в ленту задачи при каждом ответе на предложение (см. §6.2).
7. **MarkdownV2 vs HTML** — ✅ (решение Claude) **HTML mode** (`parse_mode: 'HTML'`) за простоту экранирования.
8. **Сбой отправки** — ✅ (решение Claude): логируем в `telegram_log` (см. §4), **без ретраев** (дублирование хуже потери). Индикатор «бот молчит» в UI — не сейчас, данные для него уже будут копиться в логе.

## 12. Чего НЕ делаем

- Полнотекстовое управление задачами из чата («/new купить молоко»). Соблазнительно, но это отдельная история на следующий уровень.
- Голосовые сообщения, фото, файлы.
- Групповой чат семьи. Только private 1:1 с ботом.
- Web Push как fallback. Если у пользователя нет Telegram — фича не работает; в нашем случае нас двое и у обоих есть.
